# TRADERS PROJECT: FLOW & ARCHITECTURE DOCUMENTATION

This document provides a comprehensive breakdown of the Traders project, its role hierarchy, menu functionalities, and technical blueprint.

---

## 1. Role Hierarchy: Who Adds Whom?

The project follows a cascading management hierarchy:

| Role | Responsibility | Management Action |
| :--- | :--- | :--- |
| **Super Admin** | Top-level control of the entire system. | Adds and manages **Admins**. Controls global settings. |
| **Admin** | Manages specific regions or groups of brokers. | Adds and manages **Brokers (Sub-Brokers)** and **Traders**. |
| **Broker** | Middle-layer manager for clients. | Adds and manages **Traders (Clients)**. |
| **Trader (Client)** | The end-user who places trades. | No management role; focuses on trading terminal. |

## 2. Menu Breakdown (Approx. 60 Functional States)

The menus are dynamically allocated based on roles. Below is the full list of modules and their purposes:

### A. Dashboards & Live Markets
1. **Live M2M Dashboard**: Real-time Mark-to-Market (unrealized P/L) overview for the entire hierarchy.
2. **Live M2M Detail**: Deep dive into a specific client's current profit and loss.
3. **Client Active Positions**: View specific open trades for a selected client.
4. **Broker M2M**: Simplified overview for brokers to see their downline's status.
5. **Market Watch**: Live price feed and list of scrips (stocks/commodities) for trading.
6. **Tickers**: Configure the scrolling price tickers on the main dashboard.

### B. Trade Management
7. **Active Trades**: Lists all currently running trades in the system.
8. **Closed Trades**: History of all trades that have been completed.
9. **Deleted Trades**: Audit log of trades that were manually removed by admins for correction.
10. **Group Trades**: Aggregated view of trades grouped by specific criteria.
11. **Signals**: Real-time trading signals (Buy/Sell) generated by the system or admins.
12. **Signal Admin**: Interface for Superadmin to manage/post new signals.
13. **Active Positions**: Consolidated view of all open lot sizes across scrips.
14. **Closed Positions**: Consolidated view of all finished lot sizes.

### C. User Management
15. **Trading Clients**: Master list for Manageing/Adding/Editing "Traders".
16. **Users (Sub Brokers)**: Interface to manage "Brokers" or sub-admins.
17. **Create Trading Client**: Dedicated form for onboarding new traders.
18. **Create Sub Broker**: Dedicated form for onboarding new brokers.
19. **Client Details**: Deep dive into a user's profile, balance, and ledger.
20. **Reset Client Password**: Security tool for resetting user credentials.
21. **Update Client**: Edit limits, segments (NSE/MCX), and status.
22. **Delete Client**: Tool to suspend or remove a user account.
23. **KYC Status**: Monitoring panel for Aadhar/PAN verification.

### D. Financials & Funds
24. **Trader Funds**: Overview of balances, credit limits, and current equity.
25. **Create Fund (Deposit)**: Manually credit balance to a user (Admin only).
26. **Create Fund (Withdraw)**: Manually debit balance from a user.
27. **Deposit Requests**: Queue for users requesting to add money.
28. **Withdrawal Requests**: Queue for users requesting to take money out.
29. **Negative Balance Txns**: Alerts for users who have crossed their margin limits.
30. **Accounts (Hierarchy)**: Summary of "Receivable / Payable" across the hierarchy.
31. **Broker Accounts**: Financial summary for brokers specifically.
32. **Bank Details**: Master bank settings for receiving user payments.
33. **New Client Bank**: Specific bank ID assignments for new traders.

### E. Security & Surveillance (IP Logic)
34. **IP Logins Hub**: Real-time surveillance of where everyone is logging in from.
35. **Forensic Audit Log**: Tracking browsers, OS, and location of users.
36. **Trade IP Tracking**: Tracking the specific IP address used at the moment of placing a trade.
37. **Cross-Trade Detect**: High-level alert when multiple accounts trade from the same IP (Multi-User Cluster).
38. **Banned Limit Orders**: Restrict users from placing orders far from market price (Spoofing prevention).
39. **Risk Score Monitoring**: Automated scoring based on VPN usage or rapid IP switching.

### F. Systems & Logs
40. **Action Ledger**: Audit log of every "Save", "Edit", or "Delete" action by any admin.
41. **Ip Logins Page**: Detailed log of every login attempt (Success/Fail).
42. **Notifications**: System-wide broadcasts and user-specific alerts.
43. **Global Updation**: Master switch for NSE/MCX/Options segments (Open/Close).
44. **Scrip Data**: Configuration for specific stock/commodity margins and lot sizes.
45. **Change Login Password**: Standard security update for admins/brokers.
46. **Change Transaction Password**: Security layer required for placing or deleting trades.

### G. Support & Misc
47. **Raise Ticket**: User support system for queries and issues.
48. **Support Tickets Admin**: Admin interface to reply to user queries.
49. **Voice Modulation (AI)**: Voice-activated command center for managing trades via AI.
50. **Learning Center**: Documentation and training for new users.

## 3. The IP Logic (Security Engine)

The system includes a sophisticated surveillance layer found in `IpLoginsPage` and `TradeIpTrackingPage`:

- **Multi-Account Clusters**: If multiple users login from the same IP (e.g., `192.168.1.45`), the system flags them as a "Cluster". This helps detect instances where one person is operating multiple IDs or a broker is trading on behalf of clients illegally.
- **Risk Scoring**:
    - **Residential IP**: Low Risk.
    - **Proxy/VPN Identified**: High Risk (Flagged for "Forensic Node" audit).
- **Actions**: Admins can "Terminate Cluster", "Block Access Node", or "Force KYC" on suspicious nodes.

## 6. External Mobile Trader App (Shared Backend)

The **Mobile Trader App** (located in `tradersapp/andriod_trading_a`) is built using **Expo / React Native**. 

### A. Internal Structure
- **Navigation (`AppNavigator.js`)**: Uses a Bottom Tab navigator for core sections:
    1.  **Watchlist (DashboardScreen)**: Live price tracking with sparklines.
    2.  **Trades (TradesScreen)**: Management of Pending, Active, and Closed orders.
    3.  **Portfolio (PortfolioScreen)**: Consolidated holdings and net P/L.
    4.  **Account (AccountScreen)**: Profile, Funds, and Settings.
- **Business Logic (`TradeContext.js`)**: Centralized state management for trades, prices, and notifications.

### B. Current Integration State (Alpha Prototype)
> [!IMPORTANT]
> The mobile app is currently in **Simulation Mode**.
> - **Prices**: Mocked via `setInterval` within the app.
> - **Orders**: Handled locally in memory; not yet sent to any server.
> - **Authentication**: UI-only transitions; no real token management yet.

### C. Migration Requirements
To move to production, the mobile app requires:
1.  **API Service Layer**: Migration of mock functions in `TradeContext.js` to real `fetch`/`axios` calls.
2.  **JWT Handling**: Integration with the same auth system used by the Web Admin Dashboard.
3.  **Socket Integration**: Replacing the `setInterval` simulation with a real WebSocket feed for live prices.

## 7. Deep UI Complexity & Engine Logic

Beyond the basic menus, the project contains several **"High-Level Control"** modules that manage the core trading engine logic on the frontend:

### A. Real-time Risk Management (Surveillance)
- **Auto Square Off**: Found in `useRiskManagement.js`, the system monitors M2M vs Margin live. If loss exceeds **90% of Margin**, it triggers a `DANGER` state and initiates an automatic square-off alert.
- **Risk States**: `SAFE` (Green), `WARNING` (Yellow), and `DANGER` (Red) are displayed globally in the `TopBar`.

### B. Global Batch Execution (Injection)
- **Global Updation**: Located in `GlobalUpdationPage.jsx`, this is a high-stakes 4-step wizard for batch parameter changes (Brokerage, Leverage, etc.) across 8,400+ nodes. It includes a **Simulation Phase** and an **Emergency Revert** button.

### C. Specialized Admin Utilities
- **Recalculate Brokerage**: A protected utility to force-update a client's ledger after config changes (`RecalculateBrokeragePage`).
- **Reset Account**: Specialized flow to clear client data while maintaining audit logs.
- **Security Hub**: Detection of "Multi-Account Clusters" via IP and Trade location tracking (`IpLoginsPage`).

### D. Audit & Transparency
- **Action Ledger**: A comprehensive audit trail of every admin/superadmin click, ensuring accountability for sensitive edits or deletes.
